# Agent Bridge Roundtable 可视化页面 · UI 设计需求书(设计师交接)

> 给 UI 设计师:这份文档说明**这个页面要展示什么、怎么交互、必须守住哪些技术约束**。
> 视觉风格由你主导;但第 3 节的技术栈/对接约束是**硬边界**,决定你的产出能否被接回系统,请务必遵守。
> 交付后由工程侧(我)把它接回 `skills/agent-bridge-roundtable/viz/index.html`。

---

## 1. 这是什么页面

一个**「多 Agent 圆桌审议」的实时观察台**。系统里,一个"主席" AI 会拉起 3 个左右**不同引擎的独立 AI**(席位/seat),让它们对同一个问题多轮讨论;主席调度谁发言、判断何时收敛、遇到只能人拍板的取舍会停下问用户,最后综合出方案。

**这个页面就是把这场讨论实时"放出来给人看"**:谁在说、说了什么、主席为什么这么调度、分歧在哪、什么时候在等人拍板、最终结论是什么。

一句话定位:**一个只读的、实时滚动的"审议直播 + 复盘"面板**。观众是发起这场讨论的开发者本人,他一边干别的、一边瞥一眼进展。

## 2. 谁看 / 何时看 / 想得到什么

- **观众**:发起讨论的开发者(技术背景)。**只看,不操作**——页面不改变讨论进程(所有决策在别处发生)。
- **场景**:讨论可能持续几分钟到几十分钟。观众**间歇性查看**(可能中途才打开页面,所以必须能看到已发生的历史 + 继续实时更新)。
- **想一眼得到**:①现在进行到哪轮、谁在说;②主席的**调度理由**和**分歧点**(这是审议的"智力核心",最该被看见);③**是不是在等我拍板**(要非常显眼);④结束后的**最终方案**。

## 3. 技术栈与对接硬约束(❗必读,决定能否接回去)

这个页面运行在一个**极简、零依赖、本地**的场景里,以下是不可协商的约束:

1. **最终交付物 = 单个自包含 `index.html`**。它被一个 Node 静态服务器原样吐给浏览器(`GET /`)。
   - **零外部网络请求**:不能有任何 CDN、外链字体(Google Fonts 等)、外链 CSS/JS、外链图片、外部 analytics、外部 API。服务器只绑本机、身处无外网环境,任何外链都会**加载失败**。
   - **所有资源内联**:CSS 全部 `<style>` 内联;JS 全部 `<script>` 内联;图标/插画用**内联 SVG** 或 **data-URI**;字体只能用**系统字体栈**(system-ui / Segoe UI / -apple-system / Roboto / 微软雅黑 等),不得引入 web font 文件(除非把字体 base64 内联进 CSS,但不推荐——会显著增大体积)。
   - **运行时零构建**:浏览器直接打开就能跑,不能要求运行时有打包/编译步骤。
2. **设计期你用什么随意**:Figma、手写、React/Vue 原型、Tailwind……都行。**但最终交给我的必须能编译/内联成上述单文件**。如果你用了框架/构建工具,请把它 **build 成一个自包含 HTML**(内联所有产物,去掉所有外链)再交付。推荐直接手写 **HTML + 原生 CSS + 原生 JS(ES2020)**,最省对接成本——页面逻辑很轻,不需要框架。
3. **主题**:必须同时支持**浅色 / 深色**,跟随系统 `prefers-color-scheme`。用 CSS 变量组织配色。
4. **响应式**:主要在桌面浏览器看,但要在窄窗口(如半屏 ~600px)不破版;宽内容(长文本、代码块)在自己的容器里横向滚动,页面本身不横向滚动。
5. **可访问性**:文字对比度达标;信息不能**只靠颜色**区分(席位除了颜色还要有名字/标签);字号可读(正文 ≥13px)。

> 现有的 `index.html` 已经是一个能跑的、符合以上约束的**朴素参考实现**。你可以把它当"功能已就绪、等你美化"的基线——所有数据都已正确消费,你要做的是**把它重新设计成专业、好看、信息层级清晰的版本**,而不是从零接线。

## 4. 数据从哪来(数据契约 —— 你的页面要消费这些)

页面数据全部来自服务器的两个只读接口。**你不需要实现服务器,只要按这个契约消费**:

### 4.1 实时事件流:`GET /events`(Server-Sent Events)
- 用浏览器原生 `new EventSource('/events')`。
- 连上后:服务器**先把已发生的全部事件回放一遍**(所以中途打开也能看到历史),然后**持续推送新事件**。
- 每条事件是一行 JSON,通过默认 `message` 事件到达:`es.onmessage = e => JSON.parse(e.data)`。
- **必须按 `seq` 字段去重**:`EventSource` 断线会自动重连,重连后服务器会**重新全量回放**——若不按 `seq` 去重,卡片会重复。维护一个 `Set`,`seq` 见过就跳过。
- **必须处理一个具名 `reset` 事件**:`es.addEventListener('reset', …)`。收到它表示"数据源被重置了,请**清空整个视图**(清掉去重集合和所有已渲染内容),等待新数据从头重放"。(正常讨论不会触发,是异常保护;但页面必须正确响应,否则重置后会串数据。)
- 服务器还会发**心跳注释行**(以 `:` 开头),`EventSource` 会自动忽略,你无需处理。
- 连接状态:`onopen`(已连)/ `onerror`(重连中)——请给一个**小小的连接状态指示**(如"● 实时" / "○ 重连中")。

### 4.2 按需取大文本:`GET /file?ref=<相对路径>`
- 事件里只带**摘要**;某席某轮的**完整发言**、**最终方案全文**放在文件里,事件里给一个 `outputRef` / `decisionRef` / `briefRef`(相对路径)。
- 用户想看全文时,`fetch('/file?ref=' + encodeURIComponent(ref))` 取纯文本展示(可展开/收起)。**默认不展开**,避免一屏塞满。
- 返回是 `text/plain`(Markdown 源文本)。是否渲染 Markdown 由你定(朴素展示纯文本也可接受;若渲染 MD,注意**不得引入外链库**,要内联)。

### 4.3 事件类型(内容清单的数据侧)
完整字段定义见 `skills/agent-bridge-roundtable/EVENTS.md`。这里给"每种事件 = 一块要展示的内容":

| 事件 `event` | 承载什么 | 关键字段 |
|---|---|---|
| `run:started` | 开场:议题 + 参会阵容 | `question`, `seats:[{seat,agent,model,role}]`, `maxRounds`, `budget`, `vizEnabled` |
| `round:started` | 新一轮开始 + 主席排序理由 | `round`, `mode`(parallel/serial), `plannedSpeakers`, `rationale` |
| `turn:assigned` | 指派某席发言 + 依据 | `seat`, `briefRef`, `why` |
| `turn:produced` | **某席的发言**(摘要 + 全文链接) | `seat`, `agent`, `model`, `outputRef`, `charCount`, `summary` |
| `schedule:decision` | ⭐**主席的调度决策与理由**(智力核心) | `decision`, `nextSpeakers`, `convergenceSignals`, `rationale` |
| `disagreement` | ⭐**分歧点**:各匿名立场 + 能否靠论证解决 | `topic`, `positions:[{label,stance}]`, `kind`(by-merit / needs-human) |
| `human:asked` | ⏸ **在等人拍板** | `questionId`, `question`, `options`, `why` |
| `human:answered` | 人已回答 | `questionId`, `answer` |
| `seat:skipped` | 某席掉线/超时被跳过 | `seat`, `reason` |
| `run:final` | ✅ **最终方案** | `decisionRef`, `adopted`, `tradeoffs[]`, `openIssues[]` |
| `run:terminated` | 讨论结束 | `reason` |

> ⚠️ **健壮性**:字段可能缺失(某事件没有 `rationale` 等),渲染要容错,不能因少个字段崩掉。事件**大体有序但不保证**;席位数 N **不固定**(可能 2~4)。别把"3 个席位"写死。

## 5. 信息架构与内容清单(要展示什么)

建议的结构(布局你可以重构,但这些信息都要有位置):

1. **顶部页头(常驻)**
   - 议题(`run:started.question`)。
   - **参会阵容**:每个席位一个标识,含 seat 名 + 引擎/模型(如 `codex/gpt`)+ 角色。**每席一个稳定的颜色**,全页一致(用于在时间线里辨认谁在说)——这是"独立视角"的可视化,颜色区分很重要。
   - **状态**:进行中 / 已结束(+ 原因)/ 预算(轮上限等)。
2. **⏸ 等待人类决断横幅(条件出现,要最显眼)**
   - 当有 `human:asked` 且尚无对应 `human:answered` 时,**置顶、醒目**地提示"正在等你拍板:<问题>"。这是观众最需要被打断的时刻。答复到达后收起。
3. **主时间线(核心区,按轮次分组、按时间流式增长)**
   - 按 `round` 分段(Round 0 盲发 / Round 1..收敛 / 无轮次的全局事件)。
   - 段内是**事件卡片流**,新卡片实时追加。
   - 卡片类型按第 4.3 表。**视觉层级建议**:
     - `schedule:decision`(主席决策+理由)和 `disagreement`(分歧)是**审议的智力核心 → 应最突出**(比普通发言更醒目)。
     - `turn:produced`(发言)是主体内容:显示 seat 色标 + 引擎 + 摘要 + "查看全文"(按需 `/file`)。
     - `human:asked/answered` 要有"暂停/等待"的视觉语义。
     - `seat:skipped` 是弱化的警示(掉线)。
4. **✅ 最终方案面板(结束时出现,要有仪式感)**
   - `run:final`:采纳结论 `adopted` + 权衡 `tradeoffs[]` + 未决 `openIssues[]` + "查看完整方案"(`/file` 取 `decisionRef`)。
   - 视觉上明显区别于讨论卡片(它是"结论")。

## 6. 交互与实时行为

- **实时追加**:新事件到达即插入时间线;可加**轻微入场动效**(淡入/高亮),让观众注意到"有新东西",但**别喧宾夺主/别卡顿**。
- **自动滚动**:默认可跟随最新;但如果用户**手动向上滚了**(在看历史),不要强行把他拽到底部(常见直播面板做法:显示"↓ 有新消息"按钮而非强制滚动)。这条是体验加分项,你定。
- **展开/收起全文**:`turn:produced` / `run:final` 的"查看全文"按需 `fetch /file`,就地展开、再点收起。
- **连接状态**:小指示(实时/重连中),见 4.1。
- **重置**:收到 `reset` 事件清空重来,见 4.1。

## 7. 页面状态(都要设计)

- **初始/加载**:还没收到 `run:started` 时的空态(如"等待圆桌开始…")。
- **进行中**:正常流式。
- **等待人类**:横幅置顶(第 5.2)。
- **已结束**:最终面板 + 顶部状态转为"已结束"。
- **连接断开/重连**:状态指示,不要白屏或报错弹窗。
- **重置**:清空后回到初始态等新数据。

## 8. 视觉与体验方向(你主导,但有几条"必须")

- **气质**:冷静、专业的"观测台/控制台"感,信息密度适中、可长时间盯着不累。不要花哨到干扰阅读。
- **必须**:①席位颜色编码贯穿全页且浅深色下都可辨;②`schedule:decision` / `disagreement` 视觉优先级高于普通发言;③"等待人类"状态足够抢眼;④长中文文本排版舒适(行高、字号、最大行宽)。
- **不要只靠颜色**传达含义(可访问性)。
- 图标建议用**内联 SVG**(线性、克制);emoji 也可作轻量语义标记(现基线用了 🗣️/⏸/✅ 等),你可保留或替换为 SVG。

## 9. 明确不要做(边界)

- ❌ 任何**外部网络请求**(CDN/字体/图片/analytics/API)——见第 3 节,这是硬红线。
- ❌ 任何**改变讨论的操作控件**(发送、干预、投票……)。这是**只读观察台**,不是控制台。用户拍板走的是别的通道,不在此页。
- ❌ 用户登录 / 路由 / 多页面。单页、无鉴权。
- ❌ 把席位数、轮数写死;把字段当作一定存在。
- ❌ 依赖事件"恰好一次、严格有序"——按 `seq` 去重、容错缺字段。

- **交付**:一个**自包含 `index.html`**(或能一键内联成单文件的产物)。若你用了构建工具,请附上"如何 build 成单文件"的说明,或直接给 build 好的自包含 HTML。
