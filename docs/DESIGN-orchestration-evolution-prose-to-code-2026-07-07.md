# DESIGN — 编排的演化方向:prose 调度 → 代码 driver + 模型关节

> 记录 2026-07-07 的一个架构决策讨论:dev / roundtable / **loop** 这类"多步编排"(尤其 loop 的
> 生成者↔验证者循环),顶层控制流该**每轮交给模型当主线程调度**,还是**写成一个循环脚本、用代码写死**?
>
> **结论(一句话)**:不是二选一,而是 **代码写骨架、模型当关节**;但**现在还不到写死的时候**——先让协议
> 用真实 run 定型,再把无人值守内环最先迁成代码。本文落盘的是**演化方向 + 迁移判据**,不是立即动工令。

---

## 1. 第一性原理:按"每个环节要什么能力"归属,而不是按"谁看起来更聪明"

问每一步"它真正要的是什么能力",答案自己会把活分到两边:

| 环节 | 要的是 | 该归谁 | 为什么 |
|---|---|---|---|
| 顺序执行、迭代计数、上限判断、超时轮询(send→wait 循环) | 可靠、便宜、不知疲倦 | **代码** | 模型在这里只贡献失败模式(忘 `timeout_ms`、多跑一轮、少跑一轮) |
| VERDICT 解析、逐条 AC 交叉校验 | 精确、确定性 | **代码** | 这本来就是个 parser;用自然语言给模型写 parser 规范 = 承认它该是代码 |
| 洁净树审计、代 commit、transcript 记账 | 确定性、可归因 | **代码** | 单 writer + 机械动作,零判断成分 |
| 需求挖掘→合同、缺陷争议仲裁、打回消息组织、stuck 处置、收官报告、合同中途修订 | 理解与判断 | **模型** | 每一步都要读懂语境、权衡、拿不准要升级 |

**真正的分歧点只剩一个**:顶层 `for goal / while iter` 这个循环握在谁手里。机械环节归代码是没有争议的;
争的是"谁来推进循环、谁来在每个判断点决策"。

## 2. 支持"机械内核该是代码"的证据 —— 就在本项目最近一周的工作里

这一周做的几件事,几乎每件都在给"prose 控制流不可靠"作证:

1. **VERDICT 解析要"硬化"**(只认回复首行独立一行的 `VERDICT:`、逐条结果优先于总宣称、有 fail 而首行 PASS 按
   FAIL)——给模型写这么细的解析规范,本身说明这活该是代码 parser,不是每轮让模型"看着办"。
2. **"两条致命纪律"(`wait` 必传 `timeout_ms`、别用 `wait:true`)在三个 skill 里各内联一遍**——因为 prose 调度器
   会忘,`while` 循环永远不会忘。同一条纪律复制三份 = 承认 prose 是不可靠的控制流载体。
3. **CLAUDE.md 优先级条款(2026-07-07 本轮加)是最有力的一条**:prose 驱动的控制流**可以被说服偏离**——我们花了
   一整轮去堵"模型被全局偏好文件(『目标模糊就停下讨论』『路径非最优就走更短的』)带偏、把无人值守/严格交替/
   fresh 验证者当场绕过"的口子。`for` 循环不需要这个条款,它不会被说服。**"能被 prompt 说动"正是 prose 编排的
   结构性弱点**,而它恰恰是刻意成本(防 reward-hacking)最不该被绕过的地方。
4. **圆桌已经走了这条路**:`seat-turn.mjs` / `rt-event.mjs` 就是 dogfood 发现"记账又贵又易错"之后,把机械内核
   (cp 字节直传 + 泄漏扫描 + append 事件 + 算 seq/ts/charCount)从 prose 里抽成代码工具的先例。loop 会重走同一步。
5. **主控越界(2026-07-08 补,两次独立 dogfood 实证)**:两个不同会话、不同任务的 loop run 里,主控都"顺手"
   下场干委托 agent 的活(写/跑验证基线、想直接修小缺陷),合理化话术一致(「很机械」「省上下文」「一行就能修」)。
   prose 层对策 = 白名单 + 绊线 + 收官机械自审(loop SKILL.md §主控的手);但只有 L3(driver 握循环,模型仅在
   判断点被回调、没有 Edit 工具)能**结构性消除**——这是"prose 控制流能被说服、会漂移"论点的第二组实证。

## 3. 支持"现在别写死"的证据 —— 同样在这一周

反方向的力同样强,而且指向"**时机未到**",不是"永不":

1. **协议本周改了三次**:尺子所有权/修改权、`val:script-defect` 事件、AC 设计铁规(防被考者出题)——都是本周新增
   或重写。prose 改一段话就迭代完;**代码写死后,每次协议演进都是一次开发 + 评审 + 回归**。在设计流变期写死 =
   给自己上枷锁。
2. **循环的价值恰恰藏在异常路径**:争议仲裁、`val:tainted` 重验、引擎挂掉换人、合同中途 `amended`、stuck 三选。
   这些路径现在还在**靠真跑发现**;枚举不全就写死 = 把"没想到的异常"变成死循环或崩溃。主干 happy-path 容易写死,
   但写死主干省不了多少事,难的异常分支才是模型此刻仍在兜底的部分。
3. **迁移成本已经被提前压低,不必急着现在付**:loop 的状态**全部外置**(合同 + 事件流 + git 历史 + `validation/`
   脚本),编排者本就设计成**无状态、可随时关旧开新**的角色。这意味着以后把顶层循环从模型手里换成代码 driver,
   **交接物已经是文件、不依赖会话记忆**——迁移是"换一个循环载体",不是"重写状态管理"。这笔投资已经付过了。

## 4. 决策:三级演化梯子(按信号迁移,不按日历)

```
Level 1  prose 调度(现在)
   模型当主线程,继续 dogfood,让协议在真实 run 里定型。
   ↓ 信号:某个机械步骤被反复内联/硬化(如 VERDICT 解析、记账)
Level 2  prose + 工具(下一步,低风险,随时可做)
   照 seat-turn 先例,把机械步骤抽成命令行工具:
     · verdict 解析(首行 + 逐条交叉校验,输出结构化 JSON)
     · 洁净树审计 + 以 git status 为准兜底代 commit
     · 事件记账(append transcript,算 seq/ts,校验 EVENTS.md schema)
     · "send → wait 循环 → cp textRef → 记 gen:produced" 收口封装
   控制流仍在模型手里,但每步从"几百 token 的易错操作"降为"一条命令"。
   ↓ 信号:连续 N 次真实 run 零协议改动(协议不再变)
Level 3  代码 driver + 模型关节(协议稳定后)
   把 Phase 2 内环写成 driver 脚本(bridge 是单文件,driver 可直接复用其会话机制,
   不必绕 MCP)。模型只在判断点被回调:
     · 缺陷争议仲裁      · 委托方 BLOCKED/NEEDS_CONTEXT
     · goal stuck 处置    · 合同中途修订     · 收官报告
   Phase 0/1(挖需求 → 定合同)永远归模型 —— 那是纯判断,没有循环骨架可写死。
```

**判据要具体,别凭感觉**:Level 2→3 的扳机 = **连续 N 次(建议 N≥3)真实 run 全程零协议改动**。协议还在动就停在
Level 2;动一次,N 归零重新数。

## 5. 迁移顺序:无人值守内环最先迁成代码

**最先该迁的是无人值守模式的 Phase 2 内环**,理由:

- cron 触发 + 数小时长跑 + 零人在场,是"代码握循环"的**天然场景**——没有人在旁边纠偏,prose 调度器一旦跑偏
  没人拦。
- **模型主线程自己会上下文腐化**:400k 关旧开新救的是**委托会话**;可主编排者自己串五个 goal、几十轮 send/wait,
  它自己的上下文也在膨胀,到后面判断力下降。**脚本 driver 没有这个病**——它的"上下文"就是几个文件,不腐化。
- 人在环模式可以晚一点迁,甚至长期保留 prose(有人盯着,跑偏能当场喊停,灵活性 > 确定性)。

## 6. 工程注脚:写死的循环,默认动作必须是"升级给模型",不是"重试"

即使到了 Level 3,driver 遇到**任何未识别状态**时的默认动作应该是 **halt + 升级给模型/人**,**不是重试**。
写死循环最危险的不是死板,是**在没人预料的失败上无限空转**(耗后端、耗时、无人察觉)。规则:

- 已知状态 → 按 driver 分支走。
- 未知状态 → 停,把现场(事件流尾部 + git status + 最近委托产出)交给模型判,或无人值守下 halt 并在报告标红。
- **绝不**"不认识就再跑一遍"——那是把 prose 调度器"卡住就重来"的坏习惯写进了代码,还去掉了模型"察觉不对劲"
  的能力。

## 7. 一句话总结

**你已经在正确的路上**(状态外置、seat-turn 式工具抽取,都是为这一步铺的):别现在写死;先用真实 run 把协议逼
定型(Level 1),顺手把机械步骤抽成工具(Level 2),等协议连续多次 run 不再变,再把**无人值守内环**最先变成代码
driver(Level 3),模型退到判断关节。机械环节归代码,判断环节归模型,未知状态归升级。
