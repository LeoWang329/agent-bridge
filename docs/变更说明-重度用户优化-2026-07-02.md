# Agent Bridge 变更说明：重度用户优化（2026-07-02）

> 基线：v0.9.0（commit `e375720`）。本轮改动均已合入 `main`，涉及 11 个改动点（T1–T11）加一轮整体回归修复，全部产品逻辑集中在 `scripts/agent-bridge.mjs`。

## 一、本次做了什么

起点是一份「重度用户优化建议」（`docs/PROPOSAL-heavy-user-optimizations-2026-07-01.md`）。我们没有直接照单全收，而是先让 **Codex + DeepSeek 两个独立引擎** 对建议本身做评审，筛掉「需重新设计 / 该拒绝 / 以后再做」的项，再顺藤摸瓜挖出建议里没提到的潜伏 bug，最终确定并逐项落地了下面这批改动。

执行方式（每一项都严格走了一遍）：**先实现 → 写确定性复现测试 → 跑真实后端端到端验证 → 交给 Codex 和 DeepSeek 交叉评审 → 通过后才提交下一项**。多处真实 bug 正是被这种「双引擎交叉 + 主 agent 自查」逮到的（单一评审或我自己的测试都漏过）。

按性质分成三类：

### A. 潜伏 bug 修复（P1–P5，用户已能感知的错误行为）

| 编号 | 问题 | 修复 |
|---|---|---|
| **P1 / T4** | 后端启动/运行时打到 stderr 的**良性进度信息**被当成致命 `lastError`，一条干净会话看起来像“出错了” | 新增 `lastStderr` 通道，把 OMP/Codex/Claude 三处 stderr 与真正的 `lastError` 分离；诊断信息归诊断，错误归错误 |
| **P2 / T3** | 同一个 OMP 会话并发发第二条消息会**搅乱上下文**（不像 Codex 会明确报错） | `send()` 增加在途轮次护栏：一轮未结束再发直接报错，提示先 `wait` |
| **P3 / T2** | `doctor` 的 `--version` 探测**没有超时**，某个后端卡住会拖死整个体检 | 每个探测都用异步 spawn + 定时器兜底，超时则**连整棵进程树一起 kill**（Windows 下 `codex.cmd → cmd.exe → node` 孙进程也能收掉） |
| **P4 / T5** | `clampText`（尾部截断）与 `buildSessionResult`（头部截断）方向相反，容易读错代码 | 补注释讲清两者方向相反及各自用途（本项为纯文档，无行为变化） |
| **P5 / T6** | Codex 的 `model` 参数未做统一入参校验 | 与 OMP/Claude 一致地 `sanitize`，坏 model 早报错 |

### B. 拦路的实现级基础设施（N1–N3，后续功能都依赖它）

- **N3 / T1**：`wait()` 过去用**手抄字段**拼返回值，导致 `buildSessionResult` 顶层新增的字段被静默丢弃。改成 spread 透传（只剔除 `session`/`recentEvents` 两个重字段），这样 `health`/`json`/`schemaError` 等新字段能**自动**经 `wait` 传给调用方——这是后面 A5/A1 能生效的前提。
- **N1 / T10**：路径安全原语 `resolveUnderCwd` / `readFileUnderCwd`——双端 `realpath` + `path.relative` 判定「严格在会话 cwd 内」，读文件时**开一次 fd**、`O_NOFOLLOW`、`fstat` 定 inode、开后二次校验、fd-vs-path 的 dev/ino 绑定、带上限的读循环，抗 check/use 竞态。
- **N2**：`assertString` / `assertEnum` / `assertPlainObject` 等入参校验助手，被 T7/T10/T11 复用，脏输入不再直达后端。

### C. A 档新功能（直接服务重度用户）

- **A3 / T7— 会话别名 `name`**：`open_session` 可选传人类可读别名，唯一且防与现有 id 冲突；此后凡是接受 `session_id` 的地方都能用别名。**上下文被压缩后仍能凭别名重新定位会话。**
- **A5 / T9— 派生健康度 `health`（healthy | degraded | dead）**：从「当前存活状态 + 最近一轮结果」推导，而不是看粘滞的 `lastError`。`dead`=不可用（会话关闭或进程已死，含信号杀死）；`degraded`=活着但上一轮出错；`healthy`=正在起/在跑/干净跑完后空闲。一眼判断「能不能复用这个会话」。
- **A2 / T10— `return_mode:"ref"` + `message_file`**：
  - `return_mode:"ref"`：让长评审/循环会话的产出**默认不灌进主 agent 上下文**——每轮返回 `text:null`，但保留 `charCount`/`byteCount`/`textRef`，需要细节再去读文件；显式 `max_chars` 可对单次调用覆盖。
  - `message_file`：把大提示词从**会话 cwd 内的文件**读入，而不是内联进请求（`message` 与 `message_file` 二选一）。
- **A1 / T11— Codex 原生结构化输出 `schema` → `json` / `schemaError`**：`send_message` / `open_session` 可选传 JSON Schema，转发给 Codex 原生 `outputSchema`；结果里直接带**已解析的 `json`**，或带 `schemaError:{error,rawText}` 说明为什么没拿到——**永远以数据形式返回，绝不抛异常**。仅 Codex 支持，OMP/Claude 传 `schema` 会被明确拒绝。

> 另有一项刻意**移除**：原建议里的「token 花费优化」任务已删除，并在 skill 文档里写明「使用 Agent Bridge 时不考虑 token 花费」。

## 二、整体优化围绕四条主线

1. **上下文卫生**：`return_mode:"ref"`、`message_file`、`wait` 只带轻量字段、`name` 别名——核心目的都是**让重度/长任务场景下，海量中间产出不淹没主 agent 的上下文**，同时「必读内容绝不静默丢失」（长度和 `textRef` 始终在）。
2. **可靠性与可观测性**：`health` 三态、`lastStderr` 分通道、OMP 并发护栏、`doctor` 超时——让「会话到底是好是坏、能不能复用」有明确、可编程判断的信号。
3. **安全性**：`message_file` 的路径遏制按威胁模型做了分层防护（明确定位为「防误用的安全护栏」，并诚实记录了残余边界）；入参统一校验。
4. **结构化协作**：Codex 结构化输出让「让子 agent 返回机器可读结果」成为一等能力，评审/裁决类委托可直接拿到 `json`。

## 三、会带来哪些提升

- **主 agent 上下文更耐用**：开 `return_mode:"ref"` 的评审/循环会话不再把长篇产出灌进来；大提示词走 `message_file`。长会话、多会话并行时尤其明显。
- **会话管理更省心**：`health` 一眼看出会话能否复用；`name` 别名让上下文压缩后仍可寻址；`lastStderr` 不再把良性日志误报成错误。
- **委托更稳**：OMP 并发护栏避免串轮搅乱；`doctor` 不会被单个后端拖死。
- **结构化结果开箱即用**：向 Codex 要 `json`，成功即得解析结果，失败也拿到清晰的 `schemaError`（且不会以异常形式炸掉调用链）。
- **更安全**：文件读取路径抗竞态、入参有校验，脏输入不再直达后端。

- **真实后端端到端**（`e2e-real.mjs`，跑真实 omp+codex+claude）：**26/26 通过**。

## 五、已知限制（已评审接受并记录）

- **OMP 终态事件无 id**（T3）：并发护栏基于在途标志，个别无 id 的终态事件场景下判定为「尽力而为」，已在代码注释说明。
- **Codex turn/start 失败时的 health**（T-FINAL）：turn/start 失败意味着新一轮**从未开始**，因此**刻意保留上一轮已完成的结果与健康度**（这是 T11 drift-2 契约）；该轮失败通过 `send()` 抛异常告知，真正死掉的后端由 `procGone` 判为 `dead`。强行标记会污染被保留的结果，故不改。
- **`message_file` 威胁模型**（T10）：路径遏制是「防止误读 cwd 外文件」的安全护栏，而非对抗同用户恶意进程抢换文件的硬边界（完全原子性需要 Node 标准库未提供的 per-component openat）。已把可利用窗口收窄到亚微秒级双次抢换，并诚实记录。

## 六、后续增补：上下文占用观测 `contextUsage`（A6 — 防长任务降智）

**起点问题（用户提出）**：agent-bridge 拉起的委托 agent 在**超长上下文**里会能力下降（context rot）。需要能**提前发现**、在腐化之前「关旧开新」，而不是等它变笨。经三方（Codex + DeepSeek + 主 agent）两轮定案落地。

**做了什么**：给每个会话的返回值加一个归一化的顶层字段 `contextUsage`，三后端统一形状：

```
contextUsage: { tokens, contextWindow, live, isCompacting?, autoCompactionEnabled? } | null
```

- **`tokens` 是主信号 = 当前上下文的绝对长度**。刻意**不吐百分比、也不做窗口比例的兜底**：腐化跟绝对 token 走，而各后端窗口普遍 1M，百分比会低估大窗口下的腐化（用户拍板，纯绝对 token 硬约束）。`contextWindow` 仅作**信息性**字段（窗口大小），不参与任何阈值判断。
- **`live`**：`true`=OMP 实时 `get_state` 读数；`false`=Codex/Claude 上一轮结束的快照。
- **每后端取值**：OMP `get_state.contextUsage.tokens`（实时，附 `isCompacting`/`autoCompactionEnabled`）；Codex `tokenUsage.last.inputTokens`（**非** `total`/`totalTokens`）+ `modelContextWindow`；Claude 取 `modelUsage` 中**窗口最大**的模型条目（排除小窗口 subagent；等窗口按 token 多者胜，即主对话），token = `inputTokens + cacheRead + cacheCreation`。
- **落点三处**：`summary()`（进 `status`）、`buildSessionResult()`（顶层，自动经 `wait` 的 `...extra` 透传到 `result`/`wait`）、`waitSessions().snapshot()`（`pendingSnapshots`，长会话跑到一半也能盯）。
- **`null` 语义**：尚未测到 / 该轮没吐用量，**不等于 0、不代表安全**（partial→null，绝不给半个对象）。
- **桥永不自动重开**：只吐数据；阈值（关注 300k / 重开 400k，**纯绝对 token，无窗口比例兜底**）是 **skill 编排指引**，不在产品里硬编码，重开必带「交接物」。

**评审逮到并修掉的真 bug**（Codex + DeepSeek 双引擎，两轮）：
- **F1（Claude 陈旧用量）**：`modelUsage` 原在 swallow/interrupt 闸门**之前**捕获——被吞掉的「中止残留结果」会污染当前快照；且跨轮持久，下一轮若不吐 `modelUsage` 会报上一轮的旧值。**修**：捕获移到两个提前 return 闸门**之后**（只有真正保留的结果才更新），并在 `#beginTurn` 重置 `tokenUsage`/`modelUsage`——该轮没吐用量则 `contextUsage:null`（诚实「未测」），不留旧值。
- **F2（Claude 等窗口选主）**：原「窗口最大者胜」在等窗口时按插入序取首个，可能误取 subagent。**修**：等窗口按 input 侧 token 多者胜（长上下文场景下主对话 token 远多于新起的 subagent）。同时把 `tokens:0` 统一为**有效读数**（与 OMP/Codex 一致，不再返回 null）。
- **F3（OMP list-all 陈旧 + live 语义）**：`status` 不传 id 走缓存、不刷新 `get_state`。**结论**：不强制刷新（会破坏 list-all 的同步快照不变量、并在后端卡死时拖挂）——空闲会话读数本就是当前值；正在跑的会话用 `status(id)`/`wait` 取实时。已在 `agent_bridge_status` 工具描述**写明**该契约，并**补测**了 list-all 路径。

**质量保证**：
- 确定性复现 `docs/repro-mcp-hang/repro-context-usage.mjs`（fake 三后端，**零真实消耗**）：omp 实时 + list-all + null 分支 + pendingSnapshots 快照、codex 取 `inputTokens` 且不漏 `isCompacting`、claude 选最大窗口 + token 求和 + 跨轮重置(ctxstale) + 等窗口 tie-break(ctxtie)，**8/8 通过**。
- `probe-claude-abort-fallback.mjs` 增断言：被吞的 500k 陈旧 `modelUsage` **不泄漏**（F1），**9/9 通过**。
- 真实后端端到端 `e2e-real.mjs` 增 omp/codex/claude 三处 `contextUsage` 断言，**29/29 通过**（实测：omp `live:true`+压缩标志、codex `inputTokens`+`modelContextWindow`、claude 选中 1M 主模型）。
- 双引擎两轮评审最终均 **APPROVE**、确认无遗留 bug、无回归。
