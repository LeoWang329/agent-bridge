# 匿名论证 · Round 0(盲发)

**我的立场**:通知服务用**关系型(Postgres)**作主存。

**论证**
- 通知有**稳定的 schema**:接收者、类型、渠道、状态(未读/已读/撤回)、时间戳、去重键。字段清晰、演进缓慢。
- 查询模式明确且以**关系型索引**高效覆盖:按用户拉未读列表(`user_id, status, created_at`)、按去重键幂等、按类型统计。
- **事务**天然:一次写入 + 状态机流转要原子,关系型不必额外拼装。
- 团队已有 Postgres 运维经验,**引入新存储的边际成本**(备份/监控/故障处理)不划算。

**对其它路径的预判**
- 文档型的"灵活 payload"在此**用不上**:通知 payload 可以塞进一个 `jsonb` 列,不需要为它换整套存储。
- 事件溯源能力强,但**运维与心智负担**明显更高;通知场景没到非它不可。

**需人拍板**:无(除非已读状态要求强一致 vs 最终一致,这会影响是否引入异步扇出)。
