# 匿名论证 · Round 1(回应红队 + 收敛)

**接受约束**:用户已拍板**最终一致优先**(已读状态可秒级延迟)。这削弱了我原先"强一致更省心"的隐含前提,我据此调整。

**回应 立场B 的两点反驳**
- **审计**:反驳成立。若审计是硬需求,单行就地更新确实丢历史。但我不认为要为此上完整事件溯源——用 **outbox/事件表**承载状态流转事件即可,主存仍是关系型的当前态投影。
- **扇出**:同意扇出天然异步。既然已接受最终一致,**outbox + 投递器**正是标准解,且它同时把审计事件一并记了——一张表解决两个问题。

**我的收敛方案**
- **Postgres 作主存**(当前态,团队熟、事务/索引成熟)。
- **outbox/事件表**承载状态流转事件 → 驱动多渠道扇出 + 充当审计流。
- **按用户聚合的读模型缓存**(吸收 立场C 的文档型聚合优点,但只做缓存,不作主存)。

这样既保住关系型成熟度,又拿到 立场B 要的审计与扇出解耦,**不必承担完整事件溯源的运维复杂度**。剩下是实现细节(缓存失效策略),非架构分歧。
