# 匿名论证 · Round 0(盲发)

**我的立场**:用**文档型(MongoDB)**。

**论证**
- 通知 **payload 形态多变**(不同类型字段差异大),文档型不必为每种类型改 schema。
- 读取以**用户维度聚合**:一个用户的通知列表可放进按 `user_id` 分片的集合,单次查询取回,**省 join**。
- 写多读多、**强一致需求不高**:通知不是账务,少量延迟可接受,文档型的写入路径简单、水平扩展直接。

**权衡**
- 跨用户的全局统计/复杂查询不如关系型顺手。
- 缺事务时,状态流转要靠单文档原子更新兜住(通知恰好能做到单文档聚合,风险可控)。

**需人拍板**:预期**扇出规模**(单条通知触达多少用户)——超大扇出会影响分片与写放大策略。
