# seat-b · Round 0(盲发)

**我的立场**:采用**事件溯源(Event Sourcing)**。

**论证**
- 通知的本质是**状态流转**:创建 → 投递 → 已读 → 撤回。把这些当作**不可变事件**记录,当前状态是事件的投影。
- **可审计**天然满足:"这条通知何时被谁标记已读/撤回"是产品和合规常问的问题,事件流直接回答,不必额外补审计表。
- **读模型可分离**:按用户聚合的未读列表、按类型的统计,各自投影,互不干扰写路径。

**权衡(诚实说)**
- 运维复杂度更高:事件存储 + 投影 + 重放,团队需要爬坡。
- 简单场景会显得"重"。若通知永远只是"插一行、标已读",事件溯源是杀鸡用牛刀。

**需人拍板**:是否真的需要**事件级审计**。若审计不是硬需求,事件溯源的核心优势就打折。
