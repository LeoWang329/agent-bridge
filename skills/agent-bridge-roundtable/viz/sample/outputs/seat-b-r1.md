# seat-b · Round 1(红队:反驳"关系型最稳妥")

**先 steelman 立场A**:关系型在稳定 schema + 清晰查询下确实最省心,事务与索引成熟,团队零学习成本。这是**默认正确**的选择,反对它需要给出关系型会真正踩坑的地方。

**最强失效模式(我的反驳)**
1. **审计诉求**:一旦产品要"这条通知的完整状态历史(何时投递/已读/撤回/重发)",关系型的单行 `status` 就地更新会**丢历史**。补救要加审计表 + 触发器/应用层双写——这本质是在关系型里**手搓一个残缺的事件流**,复杂度并不比事件溯源低,还更易出错(双写不一致)。
2. **多渠道扇出的最终一致**:一条通知要发 push/邮件/站内信,各渠道结果异步回收。关系型要可靠做到,通常得引入 **outbox 模式**(又是一张事件表 + 投递器)。所以"关系型 = 简单"在扇出场景并不成立。

**结论**:关系型不是错,但"它最简单"这个理由在**审计 + 扇出**面前站不住。真正的取舍是:要么上事件溯源一次性解决,要么关系型 + outbox/审计表把事件流补回来。**这更多是一致性与审计需求的取舍,不是纯技术优劣。**
